# v0.2.0 Release Checklist

This checklist survives chat-memory loss. Every step is documented here so a new session can pick up
where a previous one left off without re-deriving intent.

Human-gated steps are individually marked. **The agent must halt and wait for explicit human
approval before performing any human-gated action.**

---

## 1. Local / non-destructive validation (agent-safe, no approval needed)

Run these freely; they make no persistent changes outside the worktree.

```bash
# Full test suite — must be green before any release step
bash tests/run-all.sh

# Verify no trailing whitespace or merge artifacts
git diff --check

# Confirm working tree state — all changes should be committed
git status
```

Review `CHANGELOG.md` completeness:

- The `## [0.2.0]` header must exist.
- All merged features for 0.2.0 must be listed under the header.
- The date field (`- Unreleased`) will be updated in step 2.1 below.

---

## 2. Human-gated release steps

### 2.1 Update `CHANGELOG.md` release date

> ⚠️ **Human gate — agent must not perform this step without explicit approval.**

Change:

```
## [0.2.0] - Unreleased
```

to:

```
## [0.2.0] - <today's date, e.g. 2026-06-30>
```

### 2.2 Commit the release-date change

> ⚠️ **Human gate — agent must not perform this step without explicit approval.**

```bash
git add CHANGELOG.md
git commit -m "chore: release v0.2.0"
```

### 2.3 Create the annotated git tag

> ⚠️ **Human gate — agent must not perform this step without explicit approval.**

```bash
git tag -a v0.2.0 -m "Release v0.2.0"
```

### 2.4 Push branch and tag

> ⚠️ **Human gate — agent must not perform this step without explicit approval.**

```bash
git push origin <branch>
git push origin v0.2.0
```

### 2.5 Install the tagged release via pi

> ⚠️ **Human gate — agent must not perform this step without explicit approval.**

```bash
pi install git:git@github.com:aweiker/skills.git@v0.2.0
```

### 2.6 Update pi settings (if applicable)

> ⚠️ **Human gate — agent must not perform this step without explicit approval.**

If any pi settings reference a previous tag or path, update them now. Reload pi after changes:

```text
/reload
```

## 3. Version coherence note

`README.md` install tag, `package.json` version, and `CHANGELOG.md` header must all match.

## 4. Manual smoke test — dead-paused `/pipeline-resume`

This smoke test applies **only** to pipelines with `schema_version: 2` and `resume_supported: true`.
Pipelines with v1 status or with `resume_supported: false` / dead non-paused status are
monitor-only; they are **not** restart-resumable and must not be forced through this flow.

### 4.1 Start a resumable pipeline

Launch a pipeline that will produce at least two issues so there is a between-issues pause
opportunity. Ensure the pipeline config produces `schema_version: 2` status output.

### 4.2 Pause between issues

Use `/pipeline-pause` after the first issue completes. Wait for the pipeline to reach the
between-issues checkpoint and write `pipeline_state: "paused"` to its `status.json`.

### 4.3 Kill the paused process

Simulate a dead process by killing the pipeline shell process:

```bash
kill <pipeline-pid>
```

Verify that `status.json` still shows `pipeline_state: "paused"` (the durable pause was written before
the process slept, so the kill does not corrupt the checkpoint).

### 4.4 Run `/pipeline-resume`

```
/pipeline-resume [pipeline-id]
```

Expected behavior:

- The extension detects the process is dead (PID no longer alive).
- Because `pipeline_state: "paused"`, `schema_version: 2`, `resume_supported: true`, and the checkpoint is
  `between-issues`, the extension launches a detached `tmux` session.
- The tmux session runs `pipeline.sh --resume <status.json>`.

> **Note:** Live-paused `/pipeline-resume` is different — when the pipeline process is still alive,
> `/pipeline-resume` writes `resume` to the control file instead of launching a new tmux session.

### 4.5 Confirm tmux session

```bash
tmux list-sessions
```

A new session for the resumed pipeline should appear.

### 4.6 Confirm pipeline proceeds from `next_issue_index`

Attach to (or tail logs from) the tmux session and verify:

- The pipeline reads `next_issue_index` from `status.json` and begins at that issue.
- Issues already marked `completed` in `status.json` are **not** re-processed.
- No duplicate work is performed.

### 4.7 Confirm no duplicate completed work

After the resumed pipeline finishes, review `status.json` and any PRs/worktrees created. Each
issue from the original run should appear exactly once in the completed list.
